## 重点追问
- Q：如何解决 Agent 幻觉问题？
- A：多 agent / agent team 的思路，通过分工、交叉校验和结果仲裁来降低单个 Agent 的错误率。
- 标准答案：从模型能力、提示词约束、工具调用、检索增强、结果校验和人类兜底几个层面一起做，不能只依赖单次生成。常见做法包括：
	- 让 Agent 先检索再回答，减少无依据生成
	- 引入 planner / executor / reviewer 三段式流程
	- 对关键结论做二次校验或多模型投票
	- 对高风险输出增加规则校验、白名单和人工审核
	- 把不确定性显式暴露出来，避免“编造式自信”

## 技术八股（集中在 SW）
- Q：SW 的生命周期是怎样的？
- A：通常包括注册、安装、激活和运行四个阶段。安装阶段负责缓存静态资源，激活阶段负责接管页面并清理旧缓存，运行阶段处理网络请求和离线能力。

- Q：常见的缓存策略有哪些？
- A：常见策略有 Cache First、Network First、Stale While Revalidate、Only Cache、Only Network。前端工程里通常会根据资源类型区分：静态资源优先缓存，接口数据更偏向网络优先或带降级策略。

- Q：如果已缓存内容里有 bug 或高危漏洞，怎么补救？
- A：先做紧急止血，再做版本切换和缓存失效控制。常见手段包括更新 SW 版本、变更缓存 key、删除旧缓存、下发强制刷新策略、临时关闭离线缓存、必要时在服务端回滚有问题的静态资源。

- Q：如何从根源上解决这类问题，包括 Vite 打包等工程场景？
- A：核心是建立可回滚、可失效、可验证的发布链路。做法包括：
	- 静态资源使用内容哈希，保证文件变更即 URL 变更
	- HTML 作为入口文件尽量短缓存或不缓存，避免入口过期
	- SW 版本更新时显式清理旧缓存并做兼容处理
	- 发布时做到资源和 HTML 原子性更新，避免新旧资源混用
	- 对高风险资源增加灰度、回滚和开关控制
	- 在构建阶段加入安全扫描、依赖审计和产物校验，减少坏包进入缓存的概率
